Acton ships a built-in standard library that scripts and tests can access using the `@acton/` import prefix. When Acton initialises a project it writes the library source into a hidden `.acton/` directory at the project root; the compiler then maps every `import "@acton/..."` statement to the corresponding `.tolk` file in that directory. Because the library is plain Tolk code you can inspect, extend, or vendor it exactly like any other source file.

The library is split into two separate concerns. The **Acton stdlib** (this section) provides host-assisted helpers that only make sense inside the Acton runtime: assertion primitives, blockchain emulation, build utilities, I/O, and cryptography. The [Tolk standard library](/tolk-stdlib/overview) documents the language-level builtins that are compiled into every Tolk program — things like cell manipulation, address utilities, and TVM dictionary operations. Both layers work together, but they live in different namespaces and are imported with different prefixes.

<Note>
The `@acton/` prefix is resolved at **compile time** by the Acton CLI — it is not a package-manager import. No network access is required to use the stdlib.
</Note>

## Import Convention

Every file in the `tests/` directory and `scripts/` directory can import Acton stdlib modules with:

```tolk
import "@acton/testing/assert"
import "@acton/testing/expect"
import "@acton/emulation/testing"
import "@acton/emulation/network"
import "@acton/build"
import "@acton/crypto"
```

Contract source files in `contracts/` should **not** import `@acton/` modules — the Acton library is a host-side runtime and will not work inside TVM execution. Use the [Tolk standard library](/tolk-stdlib/overview) for on-chain logic instead.

<Tip>
Run `acton docgen` in your project root to regenerate the stdlib documentation from source at any time.
</Tip>
